    • Djeng

    a codec framework

    Project maintained by chiknriceTheme by mattgraham

    Problem

    Communication protocols doesn't end in http. As a developer for payment systems, I'm often required to integrate with switches or POS terminals which doesn't communicate using web services. When developing communications adapters for different protocols, I came to realize that there is no one solution available to cover these requirements. Some frameworks exists but are usually focused on a specific format or protocol.

    Solution

    DJENG stands for Dynamic Java Encoder (NextGen). It is the next generation of byte stream codec framework. Although originally intended to be a framework for creating ISO8583/AS2805 codecs, it evolved into a dynamic framework to easily create a codec for any message format (e.g. HSM calls, SMPP messages).

    Specifications

    These specifications are living documentation (using Concordion) which defines the functionality of every component of djeng. Automated tests are performed when generating these documentation and results are reflected at every build. A test that fails would be marked in red and passing tests would be marked in green.

    Usage

    The usual use cases involving codecs are sending and receiving messages. Sending messages involves building a message, encoding the message to a byte array using a codec, and finally sending that byte array through the wire. Receiving messages involves reading the bytes from the wire, decoding the bytes to a message using a codec and finally querying specific elements on the message.

    To build a codec, a configuration should first be created based on the message format. A message format would usually define the encoding, length, and order of every message element. With the configuration, the message codec can then be created which could be used (and re-used) for encoding and decoding messages.
